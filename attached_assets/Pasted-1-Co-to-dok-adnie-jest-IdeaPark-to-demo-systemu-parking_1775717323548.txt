1. Co to dokładnie jest

IdeaPark to demo systemu parkingowego dla osiedla / wspólnoty mieszkaniowej. Ma trzy interfejsy:

apps/mobile — aplikacja mieszkańca
apps/api — backend z logiką domenową
apps/admin — panel zarządcy / operatora

Zakres produktu:

logowanie demo JWT
dashboard mieszkańca
lista własnych pojazdów i miejsca
wyszukiwanie wolnych miejsc
rezerwowanie miejsca
udostępnianie własnego miejsca
dodawanie gości z kodem QR tekstowym
otwieranie bramy / szlabanu
panel admina z overview, miejscami, rezerwacjami, gośćmi i raportami
trwały zapis do pliku JSON, nie do bazy SQL

7. Jak działa persistence

To nie jest DB, tylko plik JSON:

ścieżka: apps/api/data/runtime-db.json
ładowany przez StoreService

Mechanika:

przy starcie tworzy folder data/, jeśli nie istnieje
jeśli runtime-db.json nie istnieje, zapisuje seed
jeśli istnieje, ładuje istniejący snapshot
każda mutacja nadpisuje cały plik

Bardzo ważne niuanse:

w archiwum plik runtime-db.json już istnieje, więc pierwsze uruchomienie czyta jego stan, a nie “czysty” seed
/reset ustawia dane na seedDatabase
seedDatabase jest tworzony z const now = new Date() przy imporcie modułu, więc czasy seedów są “zakotwiczone” do momentu startu procesu, nie do chwili resetu
nie ma transakcji, locków, optimistic concurrency, ani repozytorium DB

Shares
GET /shares/mine

Body:

  "spaceId": "string",
  "startsAt": "ISO8601",
  "endsAt": "ISO8601"
}

Logika:

jeśli space nie istnieje -> 404

Reservations
GET /reservations/mine
POST /reservations
DELETE /reservations/:id

Body POST:

{
  "spaceId": "string",
  "vehicleId": "string",
  "startsAt": "ISO8601",
  "endsAt": "ISO8601"
}

Logika create:

vehicle musi należeć do usera
space musi istnieć w tenant
blocked -> błąd
overlap z niecancelled reservation -> błąd
jeśli space private:
można tylko gdy requested range mieści się w shareWindow
wyjątek: owner może rezerwować własne miejsce bez share
status:
upcoming, gdy startsAt > now
inaczej active
zapisuje audit reservation.created

Logika cancel:

resident może anulować tylko swoją rezerwację
admin/operator mogą anulować dowolną
zmienia tylko status = 'cancelled'
dopisuje audit reservation.cancelled

Braki:

brak auto-przejścia upcoming -> active -> completed
brak walidacji startsAt < endsAt

16. Najważniejsze luki / ryzyka architektoniczne

To też jest część specyfikacji, bo ktoś odtwarzający powinien wiedzieć, co jest “cechą tego kodu”, a nie bugiem wdrożenia:

brak hashów haseł
brak refresh tokenów
brak RBAC guardów poza logiką service
/reports/summary nie ogranicza roli do admin/operator
/reset jest publiczny
brak walidacji start < end
statusy rezerwacji/gości nie są automatycznie aktualizowane
wiele modułów tworzy własne instancje StoreService, co grozi niespójnym cache w pamięci
brak shared contracts między backendem i frontendami
brak realnej integracji z bramą/szlabanem
brak Postgresa, Redis, kolejek, 2FA, billingów, webhooków i prawdziwego multi-tenant persistence — README wprost opisuje to jako starter/demo/PoC, nie pełne enterprise wdrożenie.
